## SOP 完整流程

```
1.触发 → 2.脚本取数+解析 → 3.生成HTML → 3b.QA Gate(自动) → 3c.发布短链 → 4.用户审核 → 5.定稿 → 6.测试群卡片 → 7.正式群+记录
```

### 步骤 1:调用触发
用户 @我 + 说明需求(项目名、周期)

### 步骤 2：脚本取数 + 解析
- 运行 `node scripts/fetch-project-metrics.js {project}` 拉取原始数据
- 运行 `node scripts/parse-project-metrics.js {project}` 解析为结构化 JSON
- 读取 `tmp/metrics/{project}-parsed.json`
- 数据异常(空/undefined) → 重跑脚本，连续失败再问用户
- 已自动排除当月数据、取最近6个月、计算环比

### 步骤 3:生成报告 + QA Gate + 发布
- 从 `references/html-template.html` 独立生成(禁止从其他项目复制)
- 图表从 `references/vchart-specs/*.json` 加载 spec(只替 data.values)
- **🚨 QA Gate(强制,不过不发布):**
  - 运行 `node scripts/report-qa-gate.js <生成的html文件>`
  - 退出码 0 = GATE PASSED → 继续发布
  - 退出码 1 = GATE FAILED → 修复后重跑
  - 检查内容:HTML结构/乱码/数据完整性/截断/图表配置/板块齐全(27项)
  - **断路器:最多重试 2 次(共 3 次机会),第 3 次仍 FAIL → 停止,报告用户失败项 + 请求介入**
  - **失败后强制固化闭环(禁止只写 memory/SKILL.md):**
    1. 定位根因:是生成脚本 bug 还是新失败模式?
    2. 根因在脚本 → 修脚本代码(结构性修复,不是 prompt 约束)
    3. 新失败模式 → 给 report-qa-gate.js 加对应检查项(回归防护)
    4. 在 gate 脚本头部 CHANGELOG 记录(审计可追溯)
    - 效果:代码不会忘,下次不可能再犯
- QA Gate 通过后 → 发布到 GitHub Pages → 生成 is.gd 短链
- 短链格式:`https://is.gd/{ProjectAbbr}{YYYYMM}`(无下划线)

### 步骤 4a:数据框架审(Stage 1 门禁)
- 发短链给用户,说明"请先审数据:KPI数字、图表、板块完整性"
- 通过标准:用户回复"数据OK"或等价确认
- 失败处理:修数据 → 重发短链 → 重新 Stage 1
- **数据未确认前,不进入内容审**

### 步骤 4b:内容分析审(Stage 2 门禁)
- Stage 1 通过后,提醒用户审核分析质量(核心问题、行动建议、反思问题)
- 通过标准:用户回复"定稿"
- 失败处理:修分析 → 重新 Stage 2
- **绝对禁止跳过审核**

### 步骤 5:定稿确认
- 标记正式版本(Stage 2 通过即定稿)

### 步骤 6:测试群卡片推送
- 执行 `node scripts/build-report-card.js <html> <项目> <img_key> <chat_id>`
- 或手动:截图(2x) → 上传 → 发卡片
- 测试群:oc_efda04ba9d7071b50392051a2d770447

### 步骤 7:正式群推送 + 自动记录

## 分析框架(四角度,缺一不可)

### 角度一:数据口径说明
- 统计范围(执行需求+创新,不含临时)
- 团队是否100%投入该范围
- 不适合行业基线对比时必须说明

### 角度二:多维度交叉验证

| 场景 | 单一指标结论 | 交叉验证后 |
|------|-------------|-----------|
| 人均缺陷激增 | "质量恶化" | 交付量大增,每需求密度改善 |
| 等待周期激增 | "需求前端慢" | 可能是系统产能约束 |
| 人均产出低 | "人效低下" | 口径不含全部工作 |
| 缺陷密度下降 | "质量改善" | 可能是交付量稀释 |

### 角度三:系统视角(全局产能约束)
- **瓶颈转移检查**:压缩上游后,下游能否接住?
- **WIP 与等待关系**:并行增多→排队→等待升高,是系统规律
- **用历史数据验证**产能约束假设

### 角度四:项目历史基线对比
- 从 `references/baseline-{project}.json` 加载
- 禁止用行业基线
- 6月滚动窗口均值

## 投入-产出双维度判定

| 投入(人均时长) | 产出(人均交付) | 判定 | 建议 |
|:---:|:---:|------|------|
| 高 | 高 | 超负荷 | 补充人力 |
| 低 | 高 | 效率高 | 保持 |
| 低 | 低 | 人力冗余 | 优化分工 |
| 高 | 低 | 双重问题 | 优化流程+调整人力 |

**必须用日均投入**(人均时长÷工作日),不用绝对值。

详细方法论 → `references/expert-analysis-methodology.md`

## 报告结构(17个板块)

1. 报告头部(标题+周期+日期)
2. KPI 卡片区(4指标+环比+基线差异±X带单位)
3. 特殊说明条(节假日/工作日折算)
4. 需求交付趋势图(VChart bar,6月)
5. 交付周期增长率图(正负柱状图,6月)
6. Meego操作时长趋势
7. 需求交付周期明细表(阶段拆解)
8. 跨角色协作响应时长表
9. 需求质量数据表
10. 工具效率快照(三指标同行)
11. 上月措施跟进(实际分析,不写"尚未到期")
12. 核心问题定位(三色框+隐含假设)
13. 行动计划(优化方向+责任人+时间节点)
14. 指标建议优化
15. 反思问题(2-3个)
16. 报告脚注

**尾部顺序固定**:核心问题→行动计划→指标建议→反思问题→脚注

## 数据规则(合并自铁律16-22)

| 规则 | 说明 |
|------|------|
| 排除当月 | MCP数据含当月→剔除 |
| 6月窗口 | 趋势取最近6个月(不含当月) |
| 增长率用精确值 | 用 chart_4(非chart_3整数) |
| 字段名以quota_titles为准 | 每项目不同,首次必须 get_chart_detail |
| 禁止二次计算 | MCP返回值直接用 |
| 人均操作直取 | chart_10 值,禁止自算 |
| 等待⊇间隔 | 等待周期包含跨角色间隔,不能相加 |
| 时长单位=天 | 所有Meego时长=工作日 |
| 数据校验 | 各阶段之和≈平均周期(允差2天) |

## PDCA 闭环

每月报告前必须执行:
1. **Plan**:读上月措施
2. **Do**:确认采纳状态
3. **Check**:对比数据变化
4. **Act**:有效继续/无效调整

连续2月 "pending" → 升级提醒(不静默延续)
数据持续向好 → 措施优先级可降
数据恶化 → 措施优先级升级,明确采纳或放弃
