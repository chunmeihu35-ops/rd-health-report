  const ctx = await browser.newContext({ deviceScaleFactor: 2, viewport: { width: 1280, height: 800 } });
  const page = await ctx.newPage();
  await page.goto('file:///...html');
  await page.waitForTimeout(5000);
  await page.screenshot({ path: 'output.png', fullPage: true });
  await browser.close();
})();"

> **核心教训总结**:
> 1. 卡片数据必须从报告 HTML 的 chart_spec 直接提取,禁止手动填写
> 2. 发送前必须对照检查清单自检,不能"感觉对就发"
> 3. 飞书卡片 chart 组件是纯 JSON,不支持任何 JS 函数
> 4. 颜色规则统一机械:涨=绿跌=红,不判断业务含义

**卡片 chart 组件限制**:
- 底层是 VChart,但 **不支持 JavaScript 函数**
- 所有配置必须是纯 JSON(无 function、无 formatMethod、无 formatFunc)
- 颜色映射用 seriesField + color 数组,不用 fill function
- 标签格式化:用 `"formatter": "{yField}%"` 模板语法加后缀(已验证可用)
- ❗ **formatter 字段名必须与 yField 一致**:
  - yField="growth" → formatter="{growth}%" ✓
  - yField="value" → formatter="{value}%" ✓
  - yField="growth" + formatter="{value}%" → 不显示,静默失败 ✗
- 绝不用 formatMethod/formatFunc 等函数字符串,会导致图表加载失败

**图表颜色规范(卡片+HTML 统一)**:
- 正增长 = 绿色 #4caf92
- 负增长 = 红色 #e86b6b
- 历史月 = 浅蓝 #8bb8d9
- 当月高亮 = 深蓝 #2d7eb5

> ⚠️ **VChart color 数组映射规则(铁律 2026-05-08)**:
> color 数组按 seriesField 在数据中**首次出现的顺序**映射。
> 如果数据第一条的 type="负增长",则 color 必须写 `["#e86b6b", "#4caf92"]`(红在前)。
> 如果数据第一条的 type="正增长",则 color 写 `["#4caf92", "#e86b6b"]`(绿在前)。
> **每次生成卡片时必须检查数据第一条的 type 值再决定 color 顺序。**

**KPI 颜色规则(统一 2026-05-08)**:
- 正值/上升▲ = green(无论好坏)
- 负值/下降▼ = red(无论好坏)
- 纯机械按涨跌定色,不判断业务含义

**标签位置防溢出**:
- label.position: "top",label.offset: 4
- y轴加 softMax(最大值+30%),防止标签被截断

**数据来源链接**:
- 必须使用 mcp-config.md 中存储的完整度量视图 URL
- 禁止自行拼接 URL
- URL 格式:`https://project.feishu.cn/{space}/multiProjectView/{view_id}?scope=workspaces&node={node}`

### 每月报告摘要内容

- 核心KPI(4个)+ 环比
- 周期数据 + 异常点
- 发现的问题(三类)
- 提出的措施
- 指标优化建议
- **措施跟进状态表**(下月回填)

### 版本规则

**只保留最终版本**:同一个月的报告可能有多次修改(v1→v2→v3...),但记忆里只存PMO确认后的最终版本,不存中间修改过程。

当报告有更新时,直接覆盖 `memory/projects/{project}/YYYY-MM.md`

### 报告生成前必须执行

1. **读取上月报告摘要**:`memory/projects/{project}/YYYY-MM.md`
2. **对比本月数据变化**:上月问题是否改善
3. **评估措施效果**:采纳的措施是否有正向反馈
4. **更新上月跟进状态表**

### 月度PDCA闭环

| 阶段 | 动作 | 说明 |
|------|------|------|
| Plan | 读取上月措施 | 查看上月提出了什么建议 |
| Do | 确认采纳状态 | 哪些措施被PMO采纳执行 |
| Check | 对比数据变化 | 采纳的措施数据是否改善 |
| Act | 调整或延续 | 有效则继续,无效则调整方向 |

## 数据解读规则

## 分析框架(强制)

每个核心问题的分析必须覆盖以下角度,缺一不可:

### 角度一:数据口径说明

- 明确说明统计范围(如"仅大版本执行需求")
- 团队成员是否100%投入该范围工作
- 不适合与行业基线直接对比时必须明确指出

### 角度二:多维度交叉验证

单一指标可能给出误导性结论。必须交叉验证:

| 场景 | 单一指标结论 | 交叉验证后 | 正确做法 |
|------|----------------|------------|----------|
| 人均缺陷激增 | "质量恶化" | 交付量也大增,每需求密度实际改善 | 同时看人均缺陷数+缺陷密度+交付量,并考虑工作日差异 |
| 等待周期激增 | "需求前端慢" | 可能是系统产能约束,不是局部瓶颈 | 先评估并行消化能力上限,再决定"加速上游"还是"控制WIP" |
| 人均产出低 | "人效低下" | 可能是数据口径不含全部工作 | 了解统计范围占团队总工作量的比例 |
| 缺陷密度下降 | "质量改善" | 可能是交付量稀释效应 | 绝对量+密度双看,绝对量增说明团队负担在加大 |

### 角度三:系统视角(全局产能约束)

任何局部优化建议必须考虑是否只是转移问题:

- **瓶颈转移检查**:压缩上游(如需求规划)后,下游(开发/测试)是否有足够产能接住?
- **WIP 与等待的关系**:并行需求增多→各环节排队→等待周期升高,这是系统规律而非流程缺陷
- **历史佐证**:用不同交付量下的等待周期数据验证产能约束假设

### 角度四:以项目历史基线为参照

- **禁止用行业基线做对比**(除非明确知道数据口径完全一致)
- 从 `references/baseline-{project}.json` 加载项目基线
- 对比当月数据与基线均值/范围,判断是否异常
- 示例:"人均交付0.28,高于项目6月均值0.21,处于较好水平"

### 分析框架应用示例

```
❓ 人均缺陷数上升 74.66%

✔️ 正确分析:
  - 口径:仅统计大版本需求,团队不是100%投入
  - 交叉:交付量+167%,密度-36%,工作日折算后日均+32%
  - 系统:交付量增加必然伴随缺陷绝对量增加
  - 基线:密度52低于项目均值72.43,属于较好水平
  → 结论:每需求质量改善,但绝对量压力增加,需区分缺陷来源

❌ 错误分析:
  - "人均缺陷暴增74.66%,质量风险是最突出的问题"(单一指标得结论)
  - "人均产出0.28低于行业基线0.4-0.6"(口径不同不可比)
  - "压缩需求规划加快流转"(未考虑下游产能约束)
```

> 状态:2026-03-20 升级为 A 方案(经 2 月云资源报告验证,隐含假设/场景推演效果良好)

### A方案:融入规则

**不再独立输出专家视角板块**,而是将内容融入现有板块:

| 现有板块 | 融入内容 | 说明 |
|----------|----------|------|
| **趋势图表区文字分析** | 📊 模式识别 | 历史模式+趋势预判 |
| **核心问题定位(三色框)** | ⚠️ 隐含假设 | 每个问题说明判断前提 |
| **行动计划区** | 🔮 场景推演 | 不同方案的预期影响 |
| **报告结尾** | 💡 反思问题 | 2-3个引导性问题 |

### 融入示例

**趋势分析融入模式识别**:
> "人均时长↑但人均产出↓的组合在过去6个月出现过2次,历史上通常预示人员士气/流动问题,本月需关注。"

**问题定位融入隐含假设**:
> 🚩 产出效率下降
> - 数据表现:人均交付环比-20%
> - 整改措施:xxx
> - **⚠️ 判断前提**:假设团队编制不变、无重大需求插入

**行动计划融入场景推演**:
> | 方案 | 预期影响 |
> |------|----------|
> | 维持现状 | 预计下月人均产出继续下滑10-15% |
> | 补充1名开发 | 人均时长降至合理区间,但需2-4周ramp up |
> | 砍掉20%低优需求 | 可保质量但影响业务目标达成 |

### Gaming识别(Goodhart's Law应用)

在分析时对以下情况保持警觉:

| 指标表现 | 可能的问题 | 验证动作 |
|----------|-----------|----------|
| 人均产出突增>30% | 需求拆分过细/估点虚高 | 对比需求平均规模 |
| 交付周期大幅缩短 | 跳过评审/测试缩水 | 对比缺陷密度 |
| 缺陷数突降 | 不录入/降级处理 | 询问线上问题情况 |

### 📣 项目方质疑思考(2026-03-30新增)

> 报告不是自嗨,是要说服项目方。专家解读里必须预判质疑、提供应对思路。

**核心理念**:
- 指标不是圣经,项目方有权质疑
- 报告的说服力 = 数据 + 对质疑的预判和回应
- 目的:引导 PMO 向项目方汇报的方向

**融入位置**:在「指标建议优化模块」或对应指标分析中

**必须覆盖的质疑角度**:

| 角度 | 核心问题 | 报告中如何体现 |
|------|----------|----------------|
| 指标定义合理性 | 这个指标能反映真实问题吗? | 说明指标设计意图和边界 |
| 数据可信度 | 数据准确吗?有没有统计口径问题? | 标注数据来源和已知局限 |
| 指标联动悖论 | 两个指标联合看是否有矛盾? | 明确说明悖论场景和解读方式 |
| 可操作性 | 看了这个指标我能做什么决策? | 给出具体可落地的行动建议 |

**典型质疑预判模板**:

```markdown
**🤔 项目方可能的质疑**

| 质疑点 | 预判问题 | 应对思路 |
|--------|----------|----------|
| 人均交付数量 | "投入4人比2人显示效率更低,这不合理" | 该指标需结合需求复杂度看,人多可能因为需求本身更复杂,建议增加需求规模指数辅助判断 |
| 需求颗粒度 | "大需求和小需求都算1个,不公平" | 已规划用代码量作为规模系数(待验证),当前建议对比历史同类需求 |
| 一次验收通过率 | "100%不能说明质量好" | 认同,该指标参考价值有限,质量以缺陷密度为主 |

**本报告指标局限性说明**:
(针对当前报告数据,说明哪些结论可能被质疑、如何回应)
```

**悖论场景识别(必须在报告中主动说明)**:

| 悖论场景 | 表面数据 | 实际可能 | 报告中应说明 |
|----------|----------|----------|--------------|
| 人多效率"低" | 4人完成10需求,人均2.5 | 需求复杂度高,4人是合理投入 | "人均产出下降但需求规模/复杂度上升,属于合理投入" |
| 时长高产出也高 | 人均时长↑,人均交付↑ | 超负荷运转,不可持续 | "短期高产出依赖加班,需关注团队可持续性" |
| 缺陷密度低但线上问题多 | 缺陷密度0.5 | 缺陷没录入系统 | "缺陷密度需与线上反馈交叉验证" |

**说服策略四步法**(报告行文逻辑):

1. **承认边界**:这个指标能说明什么,不能说明什么
2. **交叉验证**:用其他指标佐证,不孤立下结论
3. **预判质疑**:项目方可能会问什么
4. **给出出路**:如果质疑成立,下一步怎么办(优化指标/补充数据)

**核心态度**:指标是对话工具,不是审判工具。报告目的是发现问题、推动改进,不是给项目方"定罪"。

### 行业数据支撑(专家解读专业性)

> 来源:快手 3 年、1 万人 AI 研发范式升级复盘(InfoQ 2026-02-09)
> 用于增强专家解读的专业说服力,不修改指标体系

### 2025 DORA AI辅助软件开发报告(2026-03-30新增)

> 来源:DORA《2025 AI辅助软件开发现状》(近5000名技术专业人士调研)
> 云文档:https://www.feishu.cn/docx/Thn5dnf2Rop5w4xj1macaNYZnGd

**核心洞察:AI是"放大器"**
- AI 既放大高绩效组织的优势,也放大低绩效组织的问题
- AI 投资的最大回报不是工具本身,而是**底层组织系统**(平台质量、工作流清晰度、团队步调一致)

**关键数据**:

| 指标 | 数据 | 解读应用 |
|------|------|----------|
| AI采用率 | **90%** | AI应用已是行业标配,不用就落后 |
| 认为AI提升效率 | **80%+** | 个人感知明显 |
| 对AI生成代码不信任 | **30%** | 验证技能仍是刚需 |
| AI智能体使用率 | **39%**从不用 | 智能体模式仍在早期 |

**AI对关键指标的影响(双刃剑)**:

| 正向影响 | 效应值 |
|----------|--------|
| 个人效能 | ↑ 0.15-0.20 |
| 组织绩效 | ↑ 0.10-0.15 |
| 高价值工作时间 | ↑ 0.08-0.12 |
| 代码质量 | ↑ 0.06-0.10 |
| 产品表现 | ↑ 0.04-0.08 |
| 交付吞吐量 | ↑ 0.02-0.06 |

| 负向/中性影响 | 效应值 |
|---------------|--------|
| **交付不稳定性** | ↑ 0.10(⚠️ 核心风险) |
| 工作倦怠 | 无改善 |
| 工作摩擦 | 无改善 |

**专家解读应用场景**:

| 报告场景 | DORA数据支撑的解读 |
|----------|-------------------|
| 交付量↑但缺陷也↑ | "符合DORA发现:AI提升吞吐量但增加不稳定性,组织系统需演进以安全管控AI加速的开发" |
| 个人效率↑但团队交付无变化 | "DORA证实:AI对个人效能影响(0.15-0.20)远大于团队绩效(0.00-0.04),需设计传导机制" |
| AI试点效果不明显 | "DORA发现:AI成功取决于7项基础能力,而非工具本身。建议评估数据生态和平台质量" |

**DORA AI能力模型(7项关键能力)**:

| 能力 | 说明 | 对产研健康的启示 |
|------|------|------------------|
| 明确的AI立场 | 组织有清晰的AI使用政策 | 有政策的团队AI效能提升更显著 |
| 健康的数据生态 | 数据高质量、易访问、高度统一 | 数据治理是AI成功的基石 |
| AI可访问内部数据 | AI工具与内部系统连接 | 孤立的AI工具效果有限 |
| 健全的版本控制 | 高频提交、善用回滚 | AI加速开发更需要安全网 |
| 小批次工作 | 变更拆分为可快速测试的单元 | 减少AI带来的不稳定性 |
| **以用户为中心** | 关注最终用户体验 | ⚠️ 不以用户为中心的团队,AI反而**负面影响绩效** |
| 高质量内部平台 | 平台工程成熟度 | 平台质量低→AI对组织绩效几乎无影响 |

**平台工程关键发现**:
- **90%** 组织已采用平台工程
- 高质量平台是AI成功的关键基石
- "如果在投资AI的同时未投入构建高质量平台,组织层面不太可能获得显著回报"

**解读应用场景**:
| 场景 | 专家解读角度 |
|------|-------------|
| 团队引入AI工具后人均产出未提升 | 这是正常现象,个人提效无法自动传导到组织提效,需专门设计传导机制 |
| AI代码生成率高但交付周期不变 | 编码只占交付周期一部分,瓶颈可能在需求理解、测试、协作等环节 |
| 管理层质疑AI投入产出比 | 可引用快手数据:需从"个人效率"视角切换到"组织效能"视角 |

**快手效能度量体系参考**(不改我们的指标,仅供解读参考):
- 北极星指标:人均交付产品需求数
- 多指标互相佐证约束,防止数据被 hack
- 通过「研发活动在线化率」发现架构问题
- 通过「需求积压率」优化评审流程

**智能化演进路线**(战略视角):
- L1 辅助(Copilot)→ L2 协同(Agent)→ L3 自主(Agentic)
- 三维度方法论:AI x 效能实践、AI x 研发平台、AI x 效能度量

## 指标体系概要

### 过程效率维度(规模+效率+质量)

| 指标 | 说明 | 联动 |
|------|------|------|
| 月度需求交付数量 | 当月验收通过的需求总数 | → 人均交付数量、人均完成时长 |
| 人均需求交付数量 | 交付数量 / 团队人数 | → 月度交付数量、人均完成时长 |
| 需求交付周期 | 创建→验收的全周期(含阶段拆解) | → 跨角色协作响应时长、一次验收通过率 |
| 需求一次验收通过率 | 首次验收即通过的比例 | → 缺陷密度、交付周期 |
| 需求交付缺陷密度 | 总缺陷数 / 验收通过需求数 | → 一次验收通过率、人均缺陷数 |
| Meego核心操作平均时长 | 需求创建/任务转交/缺陷修复的平均耗时 | → 团队人均有效操作次数 |

### 协作效率维度

| 指标 | 说明 | 联动 |
|------|------|------|
| 跨角色协作响应时长 | 产品→开发、开发→测试、测试→发布的间隔时长 | → 需求交付周期、Meego操作时长 |

### 团队能效维度

| 指标 | 说明 | 联动 |
|------|------|------|
| 团队人均有效操作次数 | 总操作次数 / 团队人数 | → Meego操作时长、人均完成时长 |
| 人均完成时长 | 总工时 / 团队人数 | → 人均需求交付数量 |
| 人均缺陷数 | 总缺陷数 / 团队人数 | → 缺陷密度、人均交付数量 |

## 投入-产出双维度判定逻辑(核心)

这是判定负载/能效的**唯一标准**:

| 场景 | 投入(人均时长) | 产出(人均交付) | 判定 | 建议 |
|------|----------------|----------------|------|------|
| 人少任务重 | 高 | 高 | 超负荷 | 补充人力 |
| 最优状态 | 低 | 高 | 效率高+负载合理 | 保持 |
| 人多效率低 | 低 | 低 | 人力冗余 | 优化分工/缩编 |
| 双重问题 | 高 | 低 | 效率低+负载重 | 优化流程+调整人力 |

**❗工作日折算规则(2026-04-29新增)**:
- 投入维度必须用**日均投入**(人均时长÷工作日)而非绝对值
- 工作日不同的月份,绝对值环比无意义
- 先算日均投入,再做投入-产出判定

## 分析角度(PMO 视角)

作为 PMO 项目管理经理,分析时关注:

1. **规模-效率闭环**:交付数量是否匹配人力投入
2. **质量双维度**:通过率 + 缺陷密度是否健康
3. **协作瓶颈定位**:哪个环节间隔时长超标
4. **周期阶段拆解**:开发/测试/等待各占多少
5. **特殊因素影响**:节假日、人员变动等

## 问题分类

报告中的核心问题分为三类,用不同颜色标识:

- 🚩 **红色边框**(issue-box.one):严重问题(如产出断崖式下跌、投入产出效率极低)
- 🚩 **橙色边框**(issue-box.two):中等问题(如周期失衡、协作瓶颈)
- 🚩 **绿色边框**(issue-box.three):需关注问题(如表面良好但隐藏风险)

- 可落地整改措施(带✓列表)

## ⚠️ 三大铁律与经验教训

> 来自实际使用中的血泪经验

### 铁律一:CSS样式布局必须保持不变

- 严格按照 HTML 模板结构生成
- 除非用户明确要求调整模板,否则不能自己"创作"样式
- 白底、浅蓝KPI卡片、水平条形图、三色问题框--这些都是固定的

### 铁律二:只替换数据

- 把模板占位符换成实际数值
- 不改变布局结构
- 模板中的 `{{占位符}}` 替换为真实数据

### 铁律三:所有分析内容都要根据实际数据 + 联动解读生成

以下部分必须根据数据生成文字分析,不能套模板:
- 🧧 特殊因素说明条(节假日影响、工作日折算)
- 📊 各趋势图下文字分析
- 📋 周期表/协作表/质量表分析
- ⏱️ Meego操作时长趋势分析
- 🚩 三类问题框(红/橙/绿)
- ✅ 整改措施列表
- 📈 行动计划区

### 铁律四:文字描述必须与数据范围逻辑自洽

- 图表展示几个月数据,文字就写"近X个月",不能多也不能少
- 必须数一下实际数据条数,不能凭感觉写
- 例:图表显示6个月(2025.09-2026.02),解读必须写"近6个月",不能写"近7个月"

### 铁律五:数据可靠性校验

生成报告前必须校验数据逻辑一致性:

**需求交付周期明细校验**:
- 校验公式:开发 + 测试 + 产品 + 等待 ≈ 平均周期
- 允许误差:1-2天(四舍五入导致)
- 超过2天误差:必须在报告中标注 ⚠️ 数据可靠性警告

**示例**:
- 2026-01:17+11+7+12=47,平均周期43,差距4天 → 需标注警告
- 正常情况:各阶段相加与平均周期差距≤2天

**处理方式**:
- 在对应数据行/单元格旁标注警告符号
- 在数据说明区域说明可能原因(数据采集问题、统计口径差异等)

### 铁律六:MCP指标直接使用,不能重命名或重复计算

**规则**:
1. MCP返回的指标名称和数值直接使用
2. 报告中的列名以MCP quota_titles为准,不能自行重命名
3. 不对MCP返回值做任何二次计算(除非明确标注为"计算值")
4. 如需补充派生指标,必须另起一列并标注计算公式

**反例**:
- ❌ MCP返回"需求交付缺陷密度"=22,标注为"缺陷总数"再除以需求数算"每需求密度"
- ✅ MCP返回"需求交付缺陷密度"=22,直接显示为"需求交付缺陷密度: 22"

**教训(2026-04-29)**:MCP字段"需求交付缺陷密度"被误解为"缺陷总数",然后自行计算了"每需求密度",双重错误。

**原则**:分析前必须先确认指标定义(见 references/metrics-definition.md),不能想当然

### 铁律七:KPI卡片颜色/角标规则(2026-03-19新增)

**规则**:纯粹基于数值正负决定颜色和方向,与业务含义无关
- **正数** → 绿色向上角标 ▲ (如:+39.86%)
- **负数** → 红色向下角标 ▼ 且必须带负号(如:▼ -45.54%)

**示例**:
- 人均完成时长增加39.86% → 绿色 ▲ 39.86%(虽然业务上是负面的)
- 人均缺陷数下降45.54% → 红色 ▼ -45.54%(虽然业务上是正面的)

**易错点**:不要根据"好坏"决定颜色,统一按正负号来

### 铁律八:报告尾部顺序固定(2026-03-19新增)

**顺序**(不可调换):
1. 核心问题定位(三类问题)
2. 专家视角
3. 行动计划
4. **指标建议优化模块**
5. **脚注**(永远最后一行)

**脚注内容**:数据来源:Meego飞书项目 · 分析周期:YYYY年M月 · 特殊因素:XXX · 本报告由PMO生成

### 铁律九:指标建议必须基于当前项目数据(2026-03-19新增)

**规则**:每次生成报告的「指标建议优化模块」必须根据当前项目的实际数据分析,不能照搬其他项目的建议

**检查清单**:
- [ ] 建议1的数据依据是当前项目的吗?
- [ ] 建议2的数据依据是当前项目的吗?
- [ ] 建议3的数据依据是当前项目的吗?
- [ ] 所有阈值判断(红灯/黄灯)是基于当前数据计算的吗?

**教训来源**:2026-03-19云资源报告直接照搬紫鸟报告的建议,被PMO发现

### 铁律十:数据核对必须仔细(2026-03-19新增)

**易错点**:
- 操作次数:70次 vs 71次(差1次也是错)
- 时长:6.6天 vs 6.8天(差0.2也是错)
- 人均操作:3.68 vs 3.74(差0.06也是错)

**规则**:所有数值必须与用户提供的原始数据严格一致,不能凭记忆或估算

### 铁律十一:数据与指标名称必须对应(2026-03-30新增)

**教训来源**:LinkFox 2月报告把"团队人均有效操作次数"的基准值(17.71)错误标注为"核心操作时长"的基准值

**易错点**:
- 工具效率快照区有多个指标并排显示
- 基准值、差异值、环比等容易张冠李戴

**规则**:
- 每个数据必须与其对应的指标名称严格匹配
- 填充数据前先确认:这个数字是哪个指标的?
- 有疑问时回看原始数据截图确认

### 铁律十五:人均完成时长必须结合工作日分析(2026-04-29新增)

**问题**:人均完成时长是绝对值指标,工作日多的月份自然偏高,不代表效率变化。

**必须执行**:
1. 计算日均投入:人均时长 ÷ 当月工作日 = 日均投入(天/工作日)
2. 在基数效应说明条中展示日均对比
3. 如果日均投入接近,必须明确指出环比增幅是工作日差异驱动
4. 投入-产出判定时,用日均投入而非绝对值对比

**示例**:
- 2月(9工作日):4.78÷9=0.53天/工作日
- 3月(21工作日):11.22÷21=0.53天/工作日
- 结论:+134.73%环比增幅完全由工作日差异驱动,非效率变化

**教训(2026-04-29)**:BrowserAct 3月报告人均时长+134.73%被标记为"严重失衡",实际日均投入完全持平,误导了分析结论。

### 铁律十六:MCP数据必须排除当月(2026-04-29新增)

**问题**:MCP返回数据包含当月(未结束),如果不排除会导致数据行错位。

**必须执行**:
1. MCP返回数据后,立即识别并剔除当月数据(如4月生成报告,剔除2026-04行)
2. 取最近6个月(不含当月)作为趋势数据
3. 校验:报告最后一行月份必须等于分析周期月份

**教训(2026-04-29)**:BrowserAct 3月报告跨角色协作数据,4月数据(3.20, 2.60)被错误放入了3月行,导致2025-12到2026-03四行全部错位。

### 铁律十七:MCP字段名以quota_titles为准(2026-04-29新增)

**问题**:映射文件手写时字段名可能与MCP实际返回不一致。

**必须执行**:
1. 新增项目时,先调用 get_chart_detail 获取真实的 quota_titles
2. 映射文件中的字段名必须与 quota_titles 严格一致
3. 每个字段必须标注 value 索引(value[0]、value[1]等)
4. 禁止猜测字段名

**教训(2026-04-29)**:映射文件用"按时交付率"映射"需求一次验收通过率",用"逾期需求数"映射"需求交付缺陷密度",字段名完全不匹配导致数据误解。

### 铁律十八:MCP数据禁止二次计算(2026-04-29新增)

**问题**:MCP返回的指标已经是计算后的结果,再次计算会产生错误数据。

**必须执行**:
1. MCP返回的指标值直接使用,不做任何二次计算
2. 报告中的列名以MCP quota_titles为准
3. 如需补充指标(如"单位缺陷密度"),必须明确标注为"计算值"并说明公式

**反例**:
- ❌ MCP返回"需求交付缺陷密度"=22,标注为"缺陷总数"再除以需求数算"每需求密度"
- ✅ MCP返回"需求交付缺陷密度"=22,直接显示为"需求交付缺陷密度: 22"

**教训(2026-04-29)**:MCP字段"需求交付缺陷密度"被误解为"缺陷总数",然后自行计算"每需求密度",导致双重错误。

### 铁律十九:增长率必须用精确值计算(2026-04-29新增)

**问题**:MCP的"平均交付周期"图表返回整数(如 chart_3),但"增长率"图表返回精确值(如 chart_4 的 9.3333)。用整数计算增长率会产生大偏差。

**必须执行**:
1. 增长率计算必须使用chart_4的精确值
2. 公式:(current - prev) / prev × 100%
3. 禁止用chart_3的四舍五入整数计算

**教训(2026-04-29)**:用整数算出+85.7%和+125.0%,实际精确值算出+76.2%和+110.0%,偏差达15个百分点。

### 铁律二十:MCP字段名是项目级别的(2026-04-29新增)

**问题**:不同项目的同一图表类型,quota_titles可能完全不同。

**示例**:
- BrowserAct 跨角色协作:"待开始->开发中间隔" / "开发→测试间隔"(2个字段)
- 紫鸟浏览器 跨角色协作:"需求规划" / "待产品设计" / "待澄清" / "产品→开发间隔" / "开发→测试间隔" / "测试→发布间隔"(6个字段)

**必须执行**:
1. 每个项目第一次生成报告时,必须调用 get_chart_detail 获取该项目的真实 quota_titles
2. 不能假设字段名与其他项目相同
3. 报告表头和列数必须根据该项目的实际字段动态生成

**教训(2026-04-29)**:映射文件写死了BrowserAct的字段名,如果直接套用到紫鸟浏览器会丢失"需求规划""待澄清""测试→发布间隔"等字段。

### 铁律二十一:行动计划时间节点必须基于报告实际生成日期(2026-04-29新增)

**问题**:报告可能因各种原因延迟生成,如果时间节点写死"N+1月第1周",可能已经过去。

**必须执行**:
1. 时间节点基于报告实际生成日期(不是分析周期月份)
2. 最早的时间节点 = 报告生成日 + 1周(给报告审核+传递的时间)
3. 禁止写已经过去的日期

**示例**:
- 3月报告于4月29日生成 → 时间节点从"5月第2周"起,不能写"4月第1周"
- 4月报告于5月6日生成 → 时间节点从"5月第3周"起,不能写"5月第1周"

**教训(2026-04-29)**:3月报告于4/29生成,行动计划写"4月第1周",早已过去3周。

### 铁律二十二:生成前必须逐项核对MCP原始数据(2026-04-29新增)

**流程**:
1. MCP取数后,先输出原始数据摘要给用户确认
2. 生成报告后,逐表核对填充值与MCP原始值是否一致
3. 特别关注:月份对齐、value索引对应、当月数据是否已排除

**检查清单**:
- [ ] 跨角色协作表:每行月份对应的value[0]和value[1]是否正确
- [ ] 需求质量表:验收率和缺陷密度是否分别对应value[0]和value[1]
- [ ] KPI卡片:当月和上月值是否正确
- [ ] 最后一行月份 = 分析周期月份(不是当月)

### 铁律二十三:截图必须用本地文件(2026-04-29新增)

**问题**:远程 CDN(githack/rawgit等)有中转页/验证页,截图会截到中转页而不是报告。

**必须执行**:
1. 用 Playwright 截取本地 HTML 文件(file:// 协议)
2. 命令:`npx playwright screenshot --full-page --wait-for-timeout 5000 "file:///path/to/report.html" output.png`

### 铁律二十九:等待周期与跨角色间隔的包含关系(2026-04-30新增)

**通用公式**:
```
等待周期 = 跨角色协作间隔合计 + 其他等待(环境/依赖/评审等)
```

**跨角色间隔**是等待周期的**明细拆解**,不是独立指标。字段名因项目而异(如有的项目是"方案设计→开发",有的是"设计→开发"),但包含关系通用。

**必须执行**:

## 协作模式:效益分析顾问(西瓜老师模式)

> 角色定位:像"西瓜老师"(PMI 全球 Future 50)一样,给 PMO 指导建议

### 发现问题时的处理方式

| 场景 | 动作 | 原则 |
|------|------|------|
| 指标定义过时 | 反馈给 PMO | 只提建议,不直接改 |
| 分析过程出现矛盾 | 同步出来讨论 | 等 PMO 确认后再更新 |
| 发现指标设计问题 | 给出优化建议 | PMO 拍板,我执行 |

### 工作流程

1. **我提出**:发现问题或优化点
2. **PMO 评估**:确认是否采纳
3. **确认后执行**:我再更新指标/技能文件

**核心原则**:不擅自改动指标体系,所有变更都等 PMO 确认

### 联动解读核心逻辑

| 联动关系 | 指标组合 |
|----------|----------|
| 投入-产出双维度(唯一判定) | 人均完成时长 + 人均交付数量 |
| 规模-效率闭环 | 月度交付 ↔ 人均交付 ↔ 人均时长 |
| 周期-协作瓶颈 | 交付周期 ↔ 跨角色协作响应时长 |
| 质量双维度 | 一次验收通过率 ↔ 缺陷密度 ↔ 人均缺陷 |
| 工具效率 | Meego操作时长 ↔ 人均有效操作次数 |

**关键**:分析不是孤立看单一指标,必须交叉验证,同一结论需多个联动指标支持

### 问题分类依据

- 🔴 **红色**:产出严重萎缩 / 投入产出极低
- 🟠 **橙色**:周期失衡 / 协作瓶颈
- 🟢 **绿色**:表面良好但有隐藏风险
