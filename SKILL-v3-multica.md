## 🔄 12 状态流程（Multica Issue 驱动）

```
状态 0: 配置检查 → 状态 1: 取数 → 状态 2: 解析 
→ 状态 3: 人工审数据 → 状态 4: HTML 生成 → 状态 5: QA Gate 
→ 状态 6: 发布 → 状态 7: 人工审内容 → 状态 8: 推送测试群 
→ 状态 9: 人工审卡片 → 状态 10: 推送正式群 → 状态 11: 收尾 → 状态 12: 已完成
```

### Multica 状态映射（6 种）

| Multica 状态 | 对应流程 | 说明 |
|-------------|---------|------|
| **Backlog** | 未开始 | Issue 创建后未分配/状态为 Backlog |
| **Todo** | 已创建，等 Agent 领取 | 分配给 Agent，状态切到 Todo |
| **In Progress** | Agent 自动执行中 | 状态 0-2（配置→取数→解析）+ 状态 4-6（HTML→QA→发布）+ 状态 8（推送测试群）+ 状态 10（推送正式群）+ 状态 11（收尾） |
| **In Review** | 人工审核节点 | 状态 3（审数据）/ 状态 7（审内容）/ 状态 9（审卡片） |
| **已阻塞 (Blocked)** | 异常中断 | MCP 连续失败 / QA Gate 3 次 FAIL / 外部依赖问题 / Agent 修复不了需要用户介入 |
| **Done** | 已完成 | 全部步骤完成 + 用户反馈已收集 |

### 状态流转规则

| 当前状态 | 完成条件 | 自动流转到 | 失败处理 |
|---------|---------|-----------|---------|
| 0: 配置检查 | 配置存在且完整 | 1: 取数 | 新项目→引导用户提供信息 |
| 1: 取数 | 脚本返回数据 | 2: 解析 | 重跑 1 次，仍失败→报告用户 |
| 2: 解析 | JSON 完整 | 3: 人工审数据 | 检查数据源，修复后重跑 |
| 3: 人工审数据 | 用户回复"数据 OK" | 4: HTML 生成 | 用户指出问题→修复→重新提交审核 |
| 4: HTML 生成 | HTML 文件生成 | 5: QA Gate | 检查模板/脚本，修复后重跑 |
| 5: QA Gate | 退出码=0 | 6: 发布 | 最多重试 2 次，仍 FAIL→报告用户 |
| 6: 发布 | GitHub Pages 部署完成 | 7: 人工审内容 | 检查部署日志，修复后重跑 |
| 7: 人工审内容 | 用户回复"定稿" | 8: 推送测试群 | 用户指出问题→修复→重新提交审核 |
| 8: 推送测试群 | 卡片发送成功 | 9: 人工审卡片 | 检查 webhook/脚本，修复后重跑 |
| 9: 人工审卡片 | 用户回复"卡片 OK" | 10: 推送正式群 | 用户指出问题→修复→重新提交审核 |
| 10: 推送正式群 | 卡片发送成功 | 11: 收尾 | 检查 webhook/脚本，修复后重跑 |
| 11: 收尾 | 基线+memory+ 记录完成 | 12: 已完成 | 手动补执行遗漏步骤 |

### 状态更新机制

每完成一个状态，Agent 自动在 Issue 评论里输出：
```
✅ 状态 X: [状态名] 完成
📋 输出：[关键结果/链接]
⏭️ 下一状态：[状态名]
⏸️ 等待：[人工审核/自动执行]
```

### 错误日志格式

每个状态失败时，Agent 自动记录：
```
❌ 状态 X: [状态名] 失败
🔍 错误：[错误信息]
 位置：[脚本/文件/行号]
🔧 建议：[修复建议]
🔄 已重试：[次数]
```

## 状态详细说明

### 状态 0: 配置检查

**自动执行**

1. 读取 `references/project-config.json`
2. 查找匹配的项目（按名称或 alias）
3. 若找到 → 加载配置，进入状态 1
4. 若未找到 → 进入新项目引导：

**新项目引导流程**：
```
Agent 在 Issue 评论里问：
"未找到项目配置，请提供以下信息：
1. 项目名称：
2. 项目类型（单选：成熟型/事务型/MVP 型）：
3. MCP 取数地址（飞书项目度量视图 URL）：
4. 项目背景（1-2 句话）：
5. 正式群 ID（飞书群 ID）："

用户回复后 → Agent 自动：
1. 解析信息
2. 通过 MCP 地址提取 view_id/node
3. 生成 shortLinkPrefix（项目名缩写）
4. 追加到 project-config.json
5. 创建 references/project-context/{alias}.md（空模板）
6. 确认配置 → 进入状态 1
```

### 状态 1-2: 取数 + 解析

**自动执行**（同 V2）

**等待用户**

Agent 输出：

## 三种 Issue 类型（Multica 治理体系）

### 类型 1：产研健康报告 Issue（用户手动触发）

**触发方式**：用户创建 Issue，标题格式 `{项目名} {YYYY}年{M}月报告`

**流程**：12 状态流转（配置检查→取数→解析→人工审数据→HTML生成→QA Gate→发布→人工审内容→推送测试群→人工审卡片→推送正式群→收尾→已完成）

**产出**：HTML 报告 + 飞书卡片 + 执行日志

---

### 类型 2：单项目 Skill 执行质量复盘 Issue（自动生成）

**触发方式**：每个产研健康报告 Issue 完成后（状态 12），Agent 自动创建

**标题格式**：`[单项目复盘] {项目名} {YYYY-MM} Skill 执行质量复盘`

**流程**：
1. 自动读取本次执行的日志（`tmp/execution-log/{YYYY-MM}-{alias}.json`）
2. 分析执行质量：耗时、失败原因、重试次数、用户评分
3. 生成复盘报告（Markdown）
4. 状态切换为 **In Review**，等待用户审核
5. 用户确认后切换为 **Done**

**产出**：单项目执行质量复盘报告（含优化建议）

**目的**：单次质检，发现单点问题，防止局部优化冲突

---

### 类型 3：月度全局 Skill 健康度报告（每月 15 日自动生成）

**触发方式**：每月 15 日自动创建 Issue

**标题格式**：`[月度治理] Skill 全局健康度报告 {YYYY-MM}`

**流程**：
1. 自动汇总上月所有执行日志（所有项目）
2. 分析全局指标：总执行次数、成功率、平均耗时、失败原因分布、用户反馈趋势
3. 识别共性问题：跨项目的重复失败、Skill 膨胀风险、优化冲突
4. 生成治理报告（Markdown）
5. 状态切换为 **In Review**，等待用户审核
6. 用户确认后切换为 **Done**

**产出**：月度全局 Skill 健康度报告（含治理建议）

**目的**：全局治理，防止 Skill 膨胀，识别优化冲突，制定月度改进计划

---

### 三种 Issue 的关系

```
每月每个项目跑报告（类型 1：12 状态流程）
        ↓ 每次跑完写执行日志
每个报告完成后 → 自动生成单项目复盘 Issue（类型 2）
        ↓ 你审核
每月 15 日 → 自动生成月度全局治理 Issue（类型 3）
        ↓ 你审核
发现问题 → 新建优化 Issue → Agent 改 Skill → 你审核 → 生效
```

**防止 Skill 膨胀的机制**：
- 类型 2（单项目复盘）：识别单点问题，避免过度优化
- 类型 3（月度治理）：识别跨项目冲突，合并重复优化，清理冗余逻辑
