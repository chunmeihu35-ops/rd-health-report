# 专家解读方法论

> 融合自 Goodhart's Law、DORA/Accelerate、AI导师框架的专家分析方法

## 一、核心原则

### 1. Goodhart's Law 警觉

**核心观点**："当一个指标成为目标，它就不再是好指标"

**在报告中的应用**：
- 识别"数据好看但可能被gaming"的情况
- 质疑单一指标的改善是否反映真实进步
- 用联动指标交叉验证，对抗指标失真

**典型场景识别**：
| 现象 | 可能的gaming行为 | 验证方法 |
|------|------------------|----------|
| 人均产出突增 | 需求拆分变细/估点虚高 | 检查需求粒度、单需求平均工时 |
| 交付周期缩短 | 跳过评审/测试缩水 | 检查缺陷密度、一次验收通过率 |
| 缺陷数下降 | 缺陷不录入/降级处理 | 检查线上问题反馈、用户投诉 |

### 2. 联动验证（对抗孤立指标）

**原则**：同一结论需多个联动指标支持

**报告中的联动组合**：
- 投入-产出：人均时长 + 人均交付（必须同时看）
- 速度-质量：交付周期 + 缺陷密度（缩短周期是否牺牲质量）
- 规模-效率：月度交付 + 人均交付（是堆人头还是真效率）

### 3. 指标张力平衡（DORA思想）

**核心观点**：速度和稳定性不是trade-off，而是正相关

**四个关键维度**：
- 吞吐量：部署频率、变更前置时间
- 稳定性：变更失败率、恢复时间

**在报告中体现**：
- 不能只追求"快"而忽略"稳"
- 高效团队应该两者都好
- 如果速度↑但质量↓，说明是假性提升

## 二、专家视角板块结构

### 📊 模式识别

**定义**：从历史数据中识别重复出现的模式，给出趋势预判

**写法模板**：
```
"[指标A+指标B]的组合模式在过去[N]个月出现过[X]次，
历史上这种模式通常预示着[后果]，
本月需重点关注[具体风险]。"
```

**示例**：
- "人均时长↑但人均产出↓的组合，在过去6个月出现过2次，均在下月出现人员流失或士气下降"
- "连续2个月缺陷密度上升+交付周期缩短，通常说明团队在'赶工期'，质量风险正在积累"

### ⚠️ 隐含假设

**定义**：明确指出当前判断基于哪些前提条件

**写法模板**：
```
本报告判断基于以下假设：
① [假设1]
② [假设2]
③ [假设3]
如上述假设不成立，需重新评估结论。
```

**常见假设类别**：
- 人员假设：团队编制保持不变、无核心人员离职
- 需求假设：无重大需求插入、范围不再扩大
- 工具假设：Meego使用规范保持一致
- 外部假设：无重大节假日影响、依赖方按时交付

### 🔮 场景推演

**定义**：给出不同决策的预期影响，帮助管理者权衡

**写法模板**：
```
场景A（维持现状）：预计[结果A]
场景B（[干预措施1]）：预计[结果B]
场景C（[干预措施2]）：预计[结果C]
```

**示例**：
- 维持现状 → 预计下月人均产出继续下滑10-15%
- 补充1名开发 → 人均时长降至合理区间，但需2-4周ramp up
- 砍掉20%低优需求 → 可保质量但影响业务目标达成

### 💡 反思问题

**定义**：提出引导管理层深度思考的问题，而非直接给答案

**设计原则**：
- 问过去：上次识别的问题，采取了什么行动？效果如何？
- 问取舍：如果资源更紧张，你会先放弃什么？
- 问假设：你觉得这个判断还缺少什么信息？

**问题池**（每期选2-3个）：
1. 上月识别的[具体问题]，本月采取了什么行动？效果如何？
2. 如果再砍10%人力，你会先牺牲哪个指标？
3. 如果下月必须保住一个指标，你选哪个？为什么？
4. 团队现在最大的瓶颈是人、流程还是工具？
5. 这个数据改善是真实进步，还是统计口径/行为变化导致的？

## 三、Gaming识别检查清单

在生成报告时，对以下情况保持警觉：

| 指标表现 | 可能的问题 | 验证动作 |
|----------|-----------|----------|
| 人均产出突增>30% | 需求拆分过细/估点虚高 | 对比需求平均规模、单需求工时 |
| 交付周期大幅缩短 | 跳过评审/测试缩水 | 对比缺陷密度、一次验收通过率 |
| 缺陷数突降 | 不录入/降级处理 | 询问线上问题、用户反馈情况 |
| 一次验收通过率=100% | 流程形同虚设 | 本指标已标注"参考价值有限" |
| Meego操作时长大降 | 使用频率下降/绕过工具 | 对比操作次数、流程执行率 |

## 六、2025 DORA AI辅助软件开发报告

> 来源：DORA《2025 AI辅助软件开发现状》（近5000名技术专业人士调研）
> 云文档：https://www.feishu.cn/docx/Thn5dnf2Rop5w4xj1macaNYZnGd

### 核心洞察：AI是“放大器”
- AI 既放大高绩效组织的优势，也放大低绩效组织的问题
- AI 投资的最大回报不是工具本身，而是底层组织系统

### 关键数据
| 指标 | 数据 | 解读应用 |
|------|------|----------|
| AI采用率 | 90% | AI应用已是行业标配 |
| 认为AI提升效率 | 80%+ | 个人感知明显 |
| 对AI生成代码不信任 | 30% | 验证技能仍是刚需 |
| AI智能体使用率 | 39%从不用 | 智能体模式仍在早期 |

### AI对关键指标的影响
| 正向 | 效应值 | 负向/中性 | 效应值 |
|------|--------|---------|--------|
| 个人效能 | ↑ 0.15-0.20 | 交付不稳定性 | ↑ 0.10（⚠️核心风险） |
| 组织绩效 | ↑ 0.10-0.15 | 工作倦怠 | 无改善 |
| 高价值工作时间 | ↑ 0.08-0.12 | 工作摩擦 | 无改善 |

### 报告中的应用场景
| 场景 | 专家解读 |
|------|----------|
| 交付量↑但缺陷也↑ | 符合DORA发现：AI提升吞吐量但增加不稳定性 |
| 个人效率↑但团队交付不变 | DORA证实：AI对个人(0.15-0.20)远大于团队(0.00-0.04) |
| AI试点效果不明显 | AI成功取决于7项基础能力，而非工具本身 |

## 七、快手 AI 研发范式升级参考

> 来源：快手 3年、1万人 AI 研发范式升级复盘（InfoQ 2026-02-09）

### 核心警示
- 用AI工具 ≠ 个人提效 ≠ 组织提效
- 快手数据：AI代码生成率30%+，个人效率↑20-40%，但组织需求交付效率基本不变
- 2025 DORA报告证实这是业界普遍问题

### 快手效能度量参考
- 北极星指标：人均交付产品需求数
- 多指标互相佐证约束，防止gaming
- 通过「研发活动在线化率」发现架构问题
- 通过「需求积压率」优化评审流程

### 智能化演进路线（战略视角）
- L1 辅助(Copilot)→ L2 协同(Agent)→ L3 自主(Agentic)
- 三维度：AI x 效能实践、AI x 研发平台、AI x 效能度量
